Detailed description

template<int BLOCK_THREADS, int ITEMS_PER_THREAD, BlockHisto256Algorithm ALGORITHM = BLOCK_BYTE_HISTO_SORT>
class cub::BlockHisto256< BLOCK_THREADS, ITEMS_PER_THREAD, ALGORITHM >

BlockHisto256 provides methods for constructing (and compositing into) 256-bin histograms from 8b data partitioned across threads within a CUDA thread block.

.
Overview
A histogram counts the number of observations that fall into each of the disjoint categories (known as bins).
For convenience, BlockHisto256 provides alternative entrypoints that differ by:
  • Complete/incremental composition (compute a new histogram vs. update existing histogram data)
Template Parameters
BLOCK_THREADSThe threadblock size in threads
ITEMS_PER_THREADThe number of items per thread
ALGORITHM[optional] cub::BlockHisto256Algorithm enumerator specifying the underlying algorithm to use (default = cub::BLOCK_BYTE_HISTO_SORT)
Algorithm
BlockHisto256 can be (optionally) configured to use different algorithms:
  1. cub::BLOCK_BYTE_HISTO_SORT. Sorting followed by differentiation. More...
  2. cub::BLOCK_BYTE_HISTO_ATOMIC. Use atomic addition to update byte counts directly. More...
Usage Considerations
  • The histogram output can be constructed in shared or global memory
  • Supports partially-full threadblocks (i.e., the most-significant thread ranks having undefined values).
  • After any operation, a subsequent __syncthreads() barrier is required if the supplied BlockHisto256::SmemStorage is to be reused or repurposed by the threadblock
Performance Considerations
  • Computation is slightly more efficient (i.e., having lower instruction overhead) for:
    • BLOCK_THREADS is a multiple of the architecture's warp size
    • Every thread has a valid input (i.e., full vs. partial-tiles)
  • See cub::BlockHisto256Algorithm for performance details regarding algorithmic alternatives

Constructs a threadblock-wide histogram in shared/global memory. Each thread contributes an array of 8b input elements.

A subsequent __syncthreads() threadblock barrier should be invoked after calling this method if the supplied smem_storage is to be reused or repurposed by the threadblock.
